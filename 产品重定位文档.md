# 产品重定位文档

## 目的

这份文档用于明确 `doc-for-agent` 当前的产品重定位方向，并作为下一阶段开发的统一基线。

它主要回答 4 个问题：

- 我们现在到底在做什么产品
- 我们下一阶段要做什么
- 当前架构有没有问题
- 当前架构会不会拖累后续发展

## 产品重定位结论

`doc-for-agent` 不应该只被理解成“生成 AGENTS 的脚本”。

它现在应当被重新定义为：

**一个面向软件仓库的文档重构与引导产品。**

它的核心任务不是简单产出 Markdown 文件，而是：

- 在用户仓库文档充分时，吸收、迁移、重构已有知识
- 在用户仓库文档很差时，从代码、配置、脚本和目录结构中生成基础项目文档
- 为 agent 生成一套可执行的操作文档
- 最终以成熟产品形态进入多个 coding agent 平台

也就是说，产品正在从：

- AGENTS 生成器

重定位为：

- **项目文档系统生成器 + agent operating docs 生成器**

## 为什么要重定位

当前方向有一个天然短板：

- 如果用户仓库本身几乎没有文档
- 或者团队根本没有维护文档的习惯
- 那我们当前“吸收旧文档并重构”的能力就很难完全发挥

这会让产品的适用范围被限制在：

- 原本已经有一定文档基础的团队

这不是一个足够大的产品面。

所以我们需要把产品目标扩成两类能力：

### 1. 文档重构

适用于：

- 旧文档很多但混乱
- 已有 flat `AGENTS`
- 有 `README/docs/specs/plan/notes` 等分散文档

### 2. 文档生成

适用于：

- 几乎没有文档
- 只有代码和配置
- 用户想从零起一套人类文档和 agent 文档

## 新的产品结构

下一阶段建议把输出能力明确分成两条：

### A. 给人看的项目文档

目标是帮助团队和维护者理解项目。

建议方向：

- `docs/overview.md`
- `docs/architecture.md`
- `docs/workflows.md`
- `docs/glossary.md`
- 视情况扩展 roadmap / status / runbook

### B. 给 agent 看的操作文档

目标是帮助 coding agent 快速建立可执行上下文。

建议继续沿用并加强：

- `AGENTS/00-entry`
- `AGENTS/01-product`
- `AGENTS/02-architecture`
- `AGENTS/03-execution`
- `AGENTS/04-memory`

### C. 最终模式

最终产品建议支持三种输出模式：

- `human`
- `agent`
- `dual`

也就是：

- 只生成人类项目文档
- 只生成 agent 文档
- 同时生成两套文档系统

## 下一阶段开发重点

### 第一条线：文档生成与重构引擎

这是产品真正的核心壁垒，优先级最高。

#### 已有基础

- 仓库扫描
- repo classification
- `initialize / migrate / refresh`
- layered `AGENTS`
- flat `AGENTS` 迁移
- supporting docs 发现
- supporting docs synthesis

#### 下一步要做的事

- 在“文档很少”的仓库中生成基础 human docs
- 在“文档很乱”的仓库中进一步做：
  - synthesis
  - dedupe
  - conflict handling
  - provenance
- 让 supporting docs 不只是“被引用”，而是被压缩成 agent 可读结论
- 让 memory 文档更接近长期记忆，而不是普通介绍页
- 让 human docs 和 agent docs 共享同一个分析引擎，但输出不同结构

#### 这一线成功后的效果

无论用户仓库处于哪种状态：

- 没文档
- 文档很薄
- 文档很乱
- 有旧 `AGENTS`

都能产出一套更清晰的项目文档系统。

### 第二条线：产品化分发

这条线决定用户是否愿意真正使用这个产品。

#### 我们明确要学习的对象

我们现在明确要学习 `ui-ux-pro-max` 的产品形态，而不是照搬它的核心能力。

我们要学习的是：

- 清晰安装入口
- 清晰平台矩阵
- 统一 CLI
- 更轻的首次使用体验
- 让用户一眼看懂如何使用

#### 下一步要做的事

- 继续保留 `pipx / pip` 入口
- 增加 `npx docagent` 或 npm 全局安装
- 把 `docagent` 继续统一成产品入口
- 扩展更多平台：
  - Copilot
  - workflow / prompt 型平台
- 增强 README、release、安装说明、平台矩阵

#### 当前分发共识

短期不走：

- Go / Rust 重写
- 单文件二进制优先

短期先走：

- Python 核心引擎
- npm / npx 分发入口
- 多平台适配

原因很清楚：

- 真正的壁垒是引擎，不是实现语言
- 当前最大增长阻力是安装门槛，不是运行性能

## 当前架构是否有问题

有问题，但不是“不能继续开发”的问题，而是“如果不整理，会影响后续放大”的问题。

### 当前架构的优点

- 引擎已经模块化
- repo analysis、builders、markdown merge 已拆开
- CLI、installer、platform templates 已分层
- 有 fixtures、snapshots、unit tests
- 已能支撑多轮快速迭代

这些说明目前架构是**能继续往前推进的**。

### 当前架构的主要问题

#### 1. 产品目标比当前目录边界跑得更快

现在仓库里同时承载了：

- skill
- generator engine
- installer product
- platform adapters
- references
- dogfooding AGENTS

这些东西逻辑上成立，但在目录认知上已经开始变重。

#### 2. `installer/assets/` 是潜在维护风险

它目前很有用，因为可以支撑安装后的自包含产品运行。

但它也带来一个风险：

- 源码一份
- 打包镜像一份

如果后面继续手工维护两套内容，长期一定会成为维护负担。

这是当前架构里最需要警惕的点之一。

#### 3. 目前还没有真正的 human docs 输出层

我们现在主要生成的是 `AGENTS/`。

如果要进入“双文档系统”阶段，当前架构还缺：

- human docs builder 层
- human / agent / dual 模式
- human docs 的模板和 refresh 策略

也就是说，产品方向已经比当前代码结构更宽了。

#### 4. 安装器已经开始接近产品，但分发层还不完整

当前已经有：

- `docagent`
- `pyproject.toml`
- installer assets

但还缺：

- npm 入口
- 平台矩阵更广
- 更成熟的 release 面

所以架构不是错，而是**还停留在 Python-first 的早期产品层**。

## 当前架构会不会影响后续开发

会有影响，但影响是“需要及早整理边界”，不是“必须推倒重来”。

### 对后续开发的不利影响主要来自三点

#### 1. 引擎和分发边界不够强

如果继续往前做，很容易出现：

- 引擎逻辑为了 installer 妥协
- installer 结构反过来污染 source-of-truth 目录

这会让后续扩平台越来越难。

#### 2. 双文档系统还没有显式建模

如果下一阶段真的做 `human / agent / dual`，现在的 builders 和 CLI 设计还不够明确。

后续如果不先定义输出模型，很容易演变成：

- 一个越来越大的 AGENTS 生成器
- 但不是一个真正的双文档系统产品

#### 3. 打包镜像同步如果没有自动化，会拖慢迭代

一旦 `installer/assets/` 和真实源目录不同步：

- 测试会混乱
- 安装产物会漂移
- 用户使用到的可能不是最新逻辑

这件事如果不提早解决，会越来越痛。

## 架构判断结论

### 目前不需要做的事

- 不需要推倒重写
- 不需要立刻改成 Go / Rust
- 不需要马上大搬家目录

### 目前必须注意的事

- 强化“引擎”和“分发”的边界
- 预留 human docs 输出层
- 规划 `human / agent / dual` 模式
- 避免长期手工维护 `installer/assets/`

## 建议的下一阶段路线

### 阶段 1：稳住引擎边界

- 明确：
  - engine
  - adapter
  - installer
  - runtime bundle
- 不让 installer 反向污染 engine

### 阶段 2：引入双文档模式

- 设计 human docs 输出模型
- 设计 `--output human|agent|dual`
- 先做最小 human docs 集合

### 阶段 3：降低分发门槛

- 保留 `pipx / pip`
- 增加 `npx docagent`
- 扩平台
- 补 release 面

### 阶段 4：清理长期维护风险

- 自动同步或自动构建 `installer/assets/`
- 让打包镜像从 source-of-truth 自动导出

## 当前协作建议

后续继续适合使用两个 worktree 并行推进：

- `codex/skill-quality`
  负责文档生成与重构引擎
- `codex/product-distribution`
  负责产品化分发与安装入口

主窗口继续负责：

- 管理 `main`
- 审核分支边界
- 跑完整验证
- 合并并推送

## 判断下一阶段是否成功

如果以后用户对产品的感觉变成下面这样，就说明重定位成功了：

- “不管我仓库有没有文档，它都能帮我起一套项目文档系统”
- “它不只是生成 AGENTS，而是真的把项目知识整理成给人和给 agent 都能用的文档”
- “我不需要先理解它内部实现语言，也能快速安装”
- “它能像一个成熟工具一样出现在不同 coding agent 平台里”

## 一句话总结

`doc-for-agent` 的下一阶段，不是继续做一个更复杂的 AGENTS 生成器。

而是把它推进成：

**一个以文档重构引擎为核心、以多平台产品分发为外壳、同时服务人类文档与 agent 文档的项目文档系统产品。**
