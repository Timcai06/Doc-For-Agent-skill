# Agent Doc Migration Blueprint

This document describes how `doc-for-agent` should evolve from a generator that can initialize or refresh `AGENTS/` into a tool that can also normalize messy, pre-existing agent documentation.

The target outcome is simple:

- regardless of whether a repository starts with no agent docs, flat agent docs, layered agent docs, or scattered docs across the repo
- after running `doc-for-agent`
- the repository should converge on one primary `AGENTS/` directory using the layered `00-entry` through `04-memory` structure

## Product Goal

`doc-for-agent` should not only create new agent docs.
It should also:

- discover existing agent-facing documentation
- choose the canonical `AGENTS/` root
- migrate useful content into the layered topology
- archive replaced structures without losing history
- make the reading order unambiguous for future agents

This turns the product from "an AGENTS generator" into "an AGENTS documentation normalization and governance tool".

## Supported Starting States

The migration system should handle these repository states:

1. No agent docs at all
2. Flat `AGENTS/` docs generated by older `doc-for-agent` runs
3. Existing layered `AGENTS/` docs
4. Mixed flat + layered `AGENTS/` docs in one root
5. Agent docs in a non-root directory
6. User-authored agent docs with custom filenames
7. Important project docs scattered across `README.md`, `docs/`, `plan/`, `specs/`, `roadmap/`, or similar folders

The output target should remain the same in every case: one layered `AGENTS/` tree at the canonical root.

## Core Principles

### One Primary Root

There should be one primary `AGENTS/` root after migration.
The tool should avoid creating parallel top-level agent roots unless the user explicitly asks for that.

### Layered Output Is The End State

Flat files are a valid input shape.
They are not the long-term output shape.

The normalized target is:

```text
AGENTS/
тФЬтФАтФА 00-entry/
тФВ   тФФтФАтФА AGENTS.md
тФЬтФАтФА 01-product/
тФЬтФАтФА 02-architecture/
тФЬтФАтФА 03-execution/
тФЬтФАтФА 04-memory/
тФФтФАтФА _archive/
```

### Preserve User Knowledge

Migration should prefer carrying forward user-authored content over replacing it with generic generated prose.
Generated content should fill gaps, tighten structure, and resolve ambiguity.

### Separate Migration From Reference

Not every useful project doc should be physically moved.

- Existing agent docs should usually be migrated or archived
- General project docs should often stay where they are and be referenced from `AGENTS/`

This keeps `AGENTS/` authoritative without rewriting the whole repository.

## State Machine

The generator should internally choose among three behaviors.

### 1. `initialize`

Use when no meaningful agent docs are found.

Behavior:

- create `AGENTS/`
- generate layered starter files
- seed them from repository analysis and major project docs

### 2. `migrate`

Use when agent-oriented docs already exist but do not match the normalized layered structure.

Behavior:

- discover existing agent doc sources
- choose the canonical root
- map old content into layered files
- archive displaced flat or legacy files
- write an entrypoint note describing the migration result

### 3. `refresh`

Use when a canonical layered `AGENTS/` tree already exists.

Behavior:

- refresh generated sections
- preserve manual blocks
- update architecture and workflow facts from the codebase
- keep memory content stable unless there is a strong reason to update it

## Discovery Architecture

Migration starts with a documentation inventory pass.

The discovery stage should collect:

- all directories named `AGENTS`
- root-level agent files such as `AGENTS.md`, `CLAUDE.md`, `agents.md`, `SKILL.md`, `PROMPT.md`
- project docs likely to contain product, architecture, execution, or memory information
- existing `doc-for-agent` markers and section formats
- likely human-maintained docs that should be preserved as source material

Suggested discovery outputs:

- `agent_roots`
- `agent_files`
- `supporting_docs`
- `legacy_flat_files`
- `existing_layered_files`
- `manual_priority_docs`

## Canonical Root Selection

The tool should select one canonical destination for normalized output.

Recommended priority:

1. Root `/AGENTS`
2. The only existing `AGENTS/` directory in the repo
3. The most complete existing `AGENTS/` directory if several exist
4. Create root `/AGENTS` when agent docs are scattered but no canonical root exists

Signals that a candidate root is already canonical:

- it contains `00-entry/AGENTS.md`
- it contains multiple `01-` to `04-` directories
- it is referenced by repository docs or current workflows
- it contains manual blocks or durable memory files

## Content Bucketing

During migration, discovered documents should be bucketed by role before rendering.

### `01-product`

Examples:

- goals
- PRD material
- user flow
- target users
- positioning

Potential source files:

- `product.md`
- `README.md`
- `docs/product/*`
- `specs/*`

### `02-architecture`

Examples:

- tech stack
- backend structure
- frontend guidelines
- compatibility constraints
- service boundaries

Potential source files:

- `architecture.md`
- `frontend.md`
- `backend.md`
- `docs/architecture/*`
- ADR-like docs

### `03-execution`

Examples:

- implementation plans
- commands
- checklists
- worktree workflows
- release steps

Potential source files:

- `workflows.md`
- `plan/*`
- `roadmap/*`
- engineering notes with command sequences

### `04-memory`

Examples:

- progress logs
- lessons learned
- current state notes
- known traps and constraints learned over time

Potential source files:

- `progress.md`
- `lessons.md`
- status logs
- human-maintained handoff notes

## Flat-To-Layered Mapping Rules

The first migration target should be older flat `doc-for-agent` output.

Recommended default mapping:

- `AGENTS/product.md`
  - `01-product/001-core-goals.md`
  - `01-product/002-prd.md`
  - `01-product/003-app-flow.md`
- `AGENTS/architecture.md`
  - `02-architecture/004-tech-stack.md`
  - `02-architecture/006-backend-structure.md`
  - `02-architecture/007-architecture-compatibility.md`
- `AGENTS/frontend.md`
  - `02-architecture/005-frontend-guidelines.md`
- `AGENTS/backend.md`
  - `02-architecture/006-backend-structure.md`
- `AGENTS/workflows.md`
  - `03-execution/008-implementation-plan.md`
- `AGENTS/README.md`
  - `00-entry/AGENTS.md`
- `AGENTS/glossary.md`
  - reference appendix or preserved archive material until a dedicated glossary strategy exists

The first version does not need perfect semantic splitting.
It should preserve important content, move it into the correct neighborhood, and let the refreshed layered files become the new primary surface.

## Archive Strategy

When migration replaces older agent files, the tool should keep a clean historical trail.

Recommended default:

```text
AGENTS/_archive/
тФФтФАтФА flat/
```

Archive behavior:

- move legacy flat files into `AGENTS/_archive/flat/`
- preserve original filenames for traceability
- add a short archive note if needed
- do not treat archived files as primary inputs for future refreshes

For supporting project docs outside `AGENTS/`:

- leave them in place
- reference them from the new layered docs when helpful
- avoid copying large repository docs into `AGENTS/` unless the user explicitly wants consolidation

## Source-Of-Truth Rules After Migration

After migration, agents should have one unambiguous reading path.

Required outcomes:

- `AGENTS/00-entry/AGENTS.md` becomes the primary entrypoint
- `01-product` through `04-memory` become the primary documentation structure
- `AGENTS/_archive/` is explicitly marked historical
- referenced docs outside `AGENTS/` are secondary source material, not the default reading order

## Refresh Behavior After Migration

Once a repo has been normalized, `refresh` should behave as if the project started layered.

Recommended behavior:

- update generated facts from current code and docs
- preserve manual blocks
- preserve memory docs more conservatively than architecture or execution docs
- avoid regenerating from archived flat files

## CLI And UX Implications

The user should not need to manually choose `initialize`, `migrate`, or `refresh` in most cases.

Recommended UX:

- keep `--mode init|refresh` for explicit control
- internally detect whether the run is really an initialize, migrate, or refresh case
- surface the decision clearly in `--explain`

Useful `--explain` additions:

- detected documentation state
- chosen canonical agent root
- migration candidates
- archive plan
- files that will be referenced rather than moved

## Implementation Phases

### Phase 1. Inventory And Detection

- add documentation inventory models
- detect flat, layered, mixed, and scattered states
- report migration intent in `--explain`

### Phase 2. Migration Scaffold

- create canonical layered root selection
- add flat-to-layered mapping helpers
- archive displaced flat files

### Phase 3. Supporting Doc Assimilation

- pull product and architecture hints from scattered docs
- reference external docs from layered output
- avoid duplicating large documents unnecessarily

### Phase 4. Strong Refresh Semantics

- treat normalized layered docs as the only primary refresh surface
- ignore archived flat docs for future generation
- preserve memory files with stricter guardrails

## Non-Goals

This migration system should not:

- relocate every project document into `AGENTS/`
- invent history that the repository does not contain
- overwrite careful human-authored docs with boilerplate
- support several equal "primary" agent roots at once

The goal is convergence, not document sprawl.
