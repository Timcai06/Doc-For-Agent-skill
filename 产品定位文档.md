# 产品定位文档

## 目的

这份文档用于明确 `doc-for-agent` 当前真正的产品定位，并回答 5 个核心问题：

- 我们到底在卖什么价值
- 用户为什么不直接让 agent 临时扫代码写文档
- 我们的核心优势和壁垒是什么
- 当前架构是否支持这个方向继续发展
- 下一阶段开发该聚焦什么

## 产品定位结论

`doc-for-agent` 不应该被定义成：

- 一个单次生成 Markdown 的脚本
- 一个只会安装 skill 的平台适配器
- 一个“帮用户 prompt 一次 agent 写文档”的包装工具

它更准确的定位应该是：

**一个面向 CLI coding-agent 工作流的项目文档系统工具。**

它的核心任务不是“一次性写出文档”，而是：

- 在文档充分时，吸收、迁移、重构已有知识
- 在文档稀缺时，从代码、配置、脚本和目录结构生成基础文档系统
- 同时生成给人使用的项目文档和给 agent 使用的操作文档
- 通过统一 CLI 进入 Claude Code、Codex、CodeBuddy、Continue、Copilot 等工作流
- 让文档可以持续 `init / doctor / refresh / migrate`，而不是一次生成后失控

一句话说：

**`doc-for-agent` 是一个把仓库知识整理成可持续文档系统的产品，而不是一次性文档生成器。**

## 为什么用户不直接让 agent 临时生成文档

这是一个必须正面回答的问题。

因为现实中用户确实可以直接在 Claude Code、Codex 或其他 agent 里说：

- 扫描这个仓库
- 帮我生成 `docs/`
- 再生成一套 `AGENTS/`

这条路对小项目、一次性试验项目是成立的。

所以单纯“让 AI 写文档”本身，不是我们的护城河。

### 直接 prompting 的问题

如果只靠临时 prompting，通常会遇到这些问题：

- 输出不可重复：不同 agent、不同会话、不同 prompt，结果差异很大
- 难以 refresh：下次再跑，很难保持结构和风格一致
- 难以 migrate：旧 flat `AGENTS`、散落文档、历史说明很难稳定吸收
- 难以治理：human docs 和 agent docs 的关系没有固定模型
- 难以复用：团队多个仓库不能走一套统一命令和结构
- 难以跨平台：Claude Code、Codex、Copilot 的使用路径不一致

所以用户不是“不能让 agent 直接写文档”，而是：

**临时写得出来，不等于长期可维护。**

## 我们的核心价值主张

### 1. 把一次性输出变成可重复的文档系统

这是最核心的价值。

我们不是只交付一份文档，而是交付一套能持续运转的工作流：

- `init`
- `doctor`
- `refresh`
- `generate`
- `migrate`

这意味着用户得到的是文档生命周期，而不是一次性产物。

### 2. 把仓库知识重构成双文档系统

我们的产品不是只生成 `AGENTS/`，也不是只生成 `docs/`。

我们要做的是两套文档同时成立：

- `docs/` 给维护者、团队成员、后续接手的人
- `AGENTS/` 给 coding agent 的执行、handoff、source-of-truth 导航

这两套文档共享分析引擎，但服务对象不同。

这比单次“帮我写 README”或“帮我写 AGENTS”更系统。

### 3. 覆盖三种真实仓库状态

我们的价值不是只在“文档好的仓库”里成立。

它要覆盖三类情况：

- 几乎没文档的仓库：从代码和配置起步生成最小文档系统
- 文档很乱的仓库：吸收、迁移、去重、重构
- 已经有 agent 文档的仓库：统一结构并继续 refresh

这决定了产品面是否足够大。

### 4. 把多平台 coding-agent 工作流收口

如果用户只在单一平台里临时 prompting，他得到的是单平台、单会话结果。

而我们的产品正在做的是：

- `docagent init --ai codex`
- `docagent init --ai claude`
- `docagent init --ai continue`
- `docagent init --ai copilot`
- `docagent init --ai all`

这让产品不只是“会写文档”，而是：

**会把文档系统接到不同 coding-agent 的真实工作流里。**

### 5. 给团队一套统一标准

这也是很重要的价值。

直接让 agent 临时写文档的问题是：

- 每个仓库结构不同
- 每次产物风格不同
- 团队无法形成统一维护方式

而 `doc-for-agent` 的目标是让团队可以在多个仓库里复用：

- 一套命令
- 一套目录心智
- 一套 refresh 模型
- 一套 agent + human docs 关系

## 我们真正的核心优势

如果要更锋利地总结，我们的优势不是“更会写文档”，而是下面这些。

### 优势一：文档生命周期，而不是单次生成

这是最该坚持的产品心智。

我们卖的是：

- 文档初始化
- 文档迁移
- 文档刷新
- 文档治理

而不是：

- 一次性让模型吐出一堆 Markdown

### 优势二：双文档系统，而不是单文档输出

我们不是单纯做：

- 一个 README 生成器
- 一个 AGENTS 生成器

而是做：

- `docs/`
- `AGENTS/`
- `human / agent / dual`

这让产品可以同时服务人和 agent。

### 优势三：面向 CLI coding-agent 场景

我们的产品不是泛文档工具。

它的使用场景非常明确：

- Claude Code
- Codex
- CodeBuddy
- Continue
- Copilot

也就是说，产品不是“谁都能用一点”，而是：

**专门为 CLI coding-agent 工作流设计。**

### 优势四：从乱文档仓库中提炼结构

普通 prompting 可以扫描仓库，但未必能持续做这些事：

- 发现 supporting docs
- 判断 `initialize / migrate / refresh`
- flat `AGENTS` 迁移
- human / agent 输出分流
- confirmed / inferred / unresolved 分层

这些能力组成的不是文风，而是系统性。

## 产品不是靠什么取胜

为了避免方向跑偏，也要明确我们不该把什么当主价值。

### 不是靠“更像人写的文风”

更自然的文风是加分项，但不是主壁垒。

大模型能力会上升，单纯文风差异很容易被追平。

### 不是靠“更个性化”

个性化有价值，但如果没有统一结构和可维护性，个性化只会加重不一致。

### 不是靠“只是多支持几个平台”

多平台适配有意义，但如果核心引擎价值不强，平台数本身也不是护城河。

## 下一阶段最该持续强化的价值

从现在开始，最值得持续强化的是这三件事：

### 1. 文档系统生命周期

继续让这些命令真正成立：

- `init`
- `doctor`
- `refresh`
- `generate`
- `migrate`

### 2. 双文档输出模型

继续让这三种输出更清楚：

- `human`
- `agent`
- `dual`

### 3. CLI coding-agent 工作流整合

继续把产品做成：

- Claude Code 用户容易开始
- Codex 用户容易开始
- CodeBuddy/Continue/Copilot 用户也容易开始

也就是：

**统一入口，分平台起步。**

## 当前架构是否支持这个定位

支持，而且还算健康，但有几个地方必须继续警惕。

### 当前架构的优点

- 引擎已经模块化
- analysis / builders / markdown / engine 已拆层
- `docagent` 开始形成统一命令面
- Python + npm 双入口已经跑通
- fixtures / snapshots / unit tests 已经足够支撑迭代

这说明当前架构不是瓶颈，仍然能继续发展。

### 当前架构的主要风险

#### 1. `installer/assets/` 双份内容长期仍是风险

它现在有效，但长期仍需要靠更严格的 sync / CI 约束控制漂移。

#### 2. README 和产品入口还在持续演化

这不是坏事，但意味着产品表达层还没完全稳定。

#### 3. human / agent / dual 的产品语义仍需继续收紧

从引擎上已经能跑，但从用户心智上还需要继续变简单。

## 下一阶段开发重点

### 第一条线：继续把输出模型产品化

目标：

- 让 `refresh` / `generate` 的 human / agent / dual 更自然
- 让 human docs 更像团队会长期保留的项目文档
- 让 messy-doc / no-doc 场景下的生成结果更稳

### 第二条线：继续把入口做短

目标：

- 让 README / quickstart / help 更像真正产品入口
- 继续降低第一次使用理解成本
- 更接近 `ui-ux-pro-max` 那种“短、硬、清晰”的安装体验

## 最终总结

如果我们把产品定义成：

- 一个帮用户 prompt 一次 agent 写文档的工具

那价值不够强，也很容易被通用模型替代。

如果我们把产品定义成：

**一个把项目知识初始化、迁移、刷新、治理为双文档系统，并能进入多个 CLI coding-agent 工作流的产品**

那这个方向就是成立的，而且有清晰壁垒。
